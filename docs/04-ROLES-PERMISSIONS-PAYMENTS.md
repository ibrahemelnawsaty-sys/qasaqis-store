## 1. مقدمة ونطاق المستند

يعرّف هذا المستند **نظام الأدوار والصلاحيات (RBAC)** الكامل لمنصة «قصص أطفال»، وقدرات لوحة الأدمن (CMS)، و**تدفقات الدفع** خطوة بخطوة، بما يشمل الحالات، والإشعارات، وقواعد الأمان.

يُبنى نظام الصلاحيات على مكتبة `spatie/laravel-permission` داخل لوحة **Filament v3**، مع تجاوز كامل للسوبر أدمن عبر `Gate::before`. كل الأسماء البرمجية الواردة أدناه جاهزة للتنفيذ المباشر في الـ Seeders والـ Policies.

> **تذكير بطبيعة المنتج:** المتجر يبيع **كتب أطفال منسّقة/مطبوعة جاهزة** (23 كتابًا في 6 أقسام) — وليست قصصًا مخصّصة باسم الطفل أو صورته. لذلك **لا توجد أي تدفقات أو صلاحيات مرتبطة بالتخصيص**؛ كل تدفقات الدفع تعامل المنتج كسلعة مطبوعة جاهزة للشحن.

## 2. مصفوفة الأدوار (Roles Matrix)

### 2.1 نظرة عامة على الأدوار

| الدور | الاسم البرمجي | الغرض | العدد المتوقع | نطاق الوصول |
|---|---|---|:--:|---|
| سوبر أدمن | `super_admin` | تحكم كامل بلا استثناء | 1–3 حسابات | كل شيء (تجاوز عبر `Gate::before`) |
| أدمن | `admin` | إدارة المحتوى/المنتجات/الطلبات دون الإعدادات الحساسة | 1–4 | واسع عدا الأدوار ومفاتيح الدفع والإعدادات التقنية |
| IT | `it` | التقنية والتكاملات ومفاتيح API والصيانة | 1–2 | تقني/نظام دون البيانات المالية التفصيلية |
| مدير طلبات | `orders_manager` | دورة حياة الطلب والشحن ومراجعة إثبات الدفع | 1–3 | الطلبات والشحن والمرتجعات فقط |
| محرر محتوى | `content_editor` | الكتب/الأقسام/الصفحات/البنرات دون التسعير الحساس والحذف | 1–5 | المحتوى فقط |
| مسوّق | `marketing` | الكوبونات والبوب أب والبنرات وSEO والإحالات | 1–3 | التسويق وSEO فقط |
| دعم مقيّد | `support` | الردود على التعليقات/المراجعات/الاستفسارات ضمن منتجات محددة | 2–8 | مقيّد بـ `allowed_product_ids` + قراءة الطلب فقط |

> **آلية التقييد بالمنتجات (Support Scope):** يُطبَّق تقييد الدعم على مستوى **منتجات محددة** عبر عمود `allowed_product_ids` (نوع JSON) على جدول `users`، وتُنفَّذ الرقابة داخل الـ Policies بجمع `hasPermissionTo()` مع فحص النطاق `in_array($model->product_id, $user->allowed_product_ids)` — وليس بتكثير الصلاحيات.

**قواعد حاكمة:**

- `super_admin` يتجاوز كل شيء عبر `Gate::before(fn ($u) => $u->hasRole('super_admin') ? true : null)` — لا تُسند له صلاحيات يدويًا، ويدعم أكثر من حساب.
- `admin` **لا** يستطيع: إدارة الأدوار، مفاتيح API، حذف/تعديل حساب سوبر أدمن، التكاملات التقنية العميقة.
- `it` يرى الطلبات **بدون** الأرقام المالية الحساسة (عرض مُقنّع) — عبر Policy تُخفي حقول `total` و`proof` وبيانات العميل الكاملة.
- `support` مقيّد بـ `allowed_product_ids`؛ أي طلب/تعليق خارج منتجاته يُرفض عبر Policy.

## 4. قدرات الـ CMS من اللوحة (بدون مبرمج)

كل ما يلي يُبنى كـ **Filament v3 Resources/Pages** بحقول Rich Editor / Repeater / Builder + مكتبة وسائط (Spatie Media Library).

| القدرة | كيف تظهر في اللوحة | من يملكها |
|---|---|---|
| **تحرير نصوص الرئيسية** | صفحة «الصفحة الرئيسية» ببلوكات قابلة للترتيب (Builder): عنوان بطل، سلايدر، أقسام مميزة، شريط ثقة، شهادات | admin, content_editor |
| **إدارة السلايدر/البنرات** | Resource «البنرات» (رفع صورة WebP + عنوان + رابط + ترتيب + تفعيل/جدولة زمنية) | admin, content_editor, marketing |
| **إضافة/تعديل كتاب وربطه** | Resource «الكتب» (العنوان، المؤلف، الرسّام، القسم، الفئة العمرية، **دار النشر**، السعر/السعر المشطوب، الوصف القصير، الوصف HTML، مخرجات التعلّم Repeater، الغلاف + معرض الصور) + حقل «الظهور في»: [الرئيسية / قسم X / مميز / الأكثر مبيعًا] | admin, content_editor |
| **دور النشر (Publishers)** | Resource «دور النشر» (سِجرة، دار الشروق، بيت الحكمة، زغلول، دار النون، رؤية للنشر، MOON، 80Fekra…) مع دار افتراضية للكتب بلا دار ظاهرة | admin, content_editor |
| **الأقسام والتصنيفات** | Resource «الأقسام» (سلوكيات ومشاعر، علمية، قصص، دينية، روايات، طفولة مبكرة) مع سحب وإفلات للترتيب — الأقسام الستة تبقى ظاهرة حتى الفارغة | admin, content_editor |
| **صفحات جديدة** | Resource «الصفحات» (slug + محتوى Rich Editor + قالب): سياسة الاستبدال، من نحن، الشحن، الأسئلة الشائعة | admin, content_editor |
| **القوائم والروابط** | Resource «القوائم» (Header/Footer) بعناصر Repeater (نص + رابط داخلي/خارجي + ترتيب + تفعيل) | admin, content_editor |
| **الكوبونات** | Resource «الكوبونات» (كود، نوع الخصم %/مبلغ، حد أدنى، تاريخ انتهاء، حد الاستخدام، شحن مجاني) | admin, marketing |
| **البوب أب (دعاية/استبيان)** | Resource «البوب أب» (نوع: إعلان/استبيان، محتوى، صورة، شرط الظهور: بعد N ثانية/عند الخروج، صفحات الاستهداف، تفعيل/جدولة) + تخزين ردود الاستبيان | admin, marketing |
| **SEO لكل صفحة** | تبويب «SEO» داخل كل Resource (title, meta description, OG image, canonical, slug, index/noindex) + إعدادات عامة (sitemap, robots, schema, بكسل التتبّع) | admin, marketing (نصوص) + it (تقني) |
| **محرك البحث والفلترة** | إعداد فهرسة البحث الاحترافي: بحث باسم الكتاب/الدار/المؤلف مع **تطبيع عربي** (همزات/تاء مربوطة/تشكيل) + فلاتر (قسم، دار نشر، فئة عمرية، سعر) + اقتراح فوري | admin, it (تقني) |
| **الوسائط** | مكتبة وسائط مركزية بتحويل WebP تلقائي متعدد الأحجام + srcset | حسب `media.*` |
| **شريط الإعلان العلوي** | إعداد نصي بسيط (نص + رابط + لون + تفعيل) | admin, marketing |

> **مبدأ:** لا شيء مما سبق يحتاج تعديل كود — كله محرَّر من الـ Resources. النصوص العربية بنبرة العلامة (مصري دافئ) قابلة للتحرير بالكامل.

## 5. تدفقات الدفع (Payment Flows)

### 5.1 طرق الدفع والحالة الافتراضية

| الطريقة | الاسم البرمجي | افتراضيًا | يتطلب مفتاح API | يعمل دائمًا؟ |
|---|---|:--:|:--:|:--:|
| بطاقات/محافظ أونلاين (Paymob/Kashier) | `online_gateway` | مفعّل إن وُجد المفتاح | ✅ | لا (يختفي بلا مفتاح) |
| إنستاباي (تحويل يدوي) | `manual_instapay` | مفعّل | ❌ | ✅ |
| فودافون كاش (تحويل يدوي) | `manual_vodafone` | مفعّل | ❌ | ✅ |
| تحويل بنكي (يدوي) | `manual_bank` | اختياري | ❌ | ✅ |
| الدفع عند الاستلام | `cod` | مفعّل | ❌ | ✅ |

**منطق الإخفاء التلقائي للدفع الأونلاين:**

```php
public function isOnlineEnabled(): bool
{
    return filled(config('services.payment_gateway.api_key'));
}
// إن غاب المفتاح: يُخفى خيار «ادفع أونلاين»، وتظهر شارة
// «الدفع الأونلاين مغلق حاليًا»، وتبقى المسارات اليدوية + COD تعمل دائمًا.
```

> **ملاحظة معمارية:** بوابة الدفع خلف `PaymentGateway interface` — يمكن تبديل Paymob بـ Kashier دون تعديل منطق الطلب. الدفع اليدوي + COD هو **المسار الافتراضي الذي يعمل دائمًا** حتى قبل تفعيل أي بوابة.

### 5.2 حالات الطلب (Order Status) — موحّدة عبر كل المسارات

| الحالة | الاسم البرمجي | المعنى |
|---|---|---|
| بانتظار الدفع | `pending_payment` | طلب أُنشئ ولم يُدفع (بوابة أونلاين لم تكتمل) |
| بانتظار إثبات الدفع | `awaiting_proof` | دفع يدوي؛ العميل لم يرفع الإثبات بعد |
| قيد مراجعة الإثبات | `proof_under_review` | رُفع الإثبات وينتظر قرار الأدمن/مدير الطلبات |
| مدفوع | `paid` | تأكد الدفع (بوابة نجحت أو إثبات مقبول) |
| بانتظار تأكيد واتساب | `awaiting_confirmation` | تأكيد الطلب قبل التجهيز (خاصة لـ COD) |
| قيد التجهيز | `processing` | جارٍ التحضير والتغليف |
| تم الشحن | `shipped` | سُلّم لشركة الشحن مع رقم تتبّع |
| تم التسليم | `delivered` | استلمه العميل (ولـ COD: حُصّل المبلغ) |
| مرفوض الدفع | `payment_rejected` | إثبات غير صالح → يُعاد للعميل لإعادة الرفع |
| ملغى | `cancelled` | أُلغي الطلب |
| مُسترد | `refunded` | استُرد المبلغ |

---

### 5.3 تدفق الدفع الأونلاين (Paymob/Kashier)

```
1. العميل يضيف الكتب للسلة ← يملأ بيانات الشحن (اسم، هاتف، عنوان، محافظة).
2. يختار «الدفع أونلاين بالبطاقة/المحفظة».
   - النظام يتحقق isOnlineEnabled() ← إن false يُخفى الخيار من الأساس.
3. إنشاء الطلب بحالة pending_payment + قفل السلة.
4. إنشاء payment intent عبر PaymentGateway interface.
5. تحويل العميل لصفحة/iframe الدفع الآمنة.
6a. نجاح الدفع ← البوابة ترسل Webhook (verify HMAC) ← الحالة = paid ← processing.
6b. فشل/إلغاء ← تبقى pending_payment ← رسالة «لم تكتمل عملية الدفع، حاول ثانية أو اختر طريقة أخرى».
7. تأكيد بريد/واتساب للعميل برقم الطلب.
```

> **نقاط حرجة:** التحقق من الـ Webhook بالتوقيع (HMAC) لا من ردّ المتصفح فقط؛ Idempotency لمنع ازدواج التأكيد؛ تسجيل كل محاولة في جدول `payment_transactions`.

---

### 5.4 تدفق الدفع اليدوي (إنستاباي / فودافون كاش / بنكي)

```
1. العميل يختار طريقة يدوية (مثلاً فودافون كاش).
2. إنشاء الطلب بحالة awaiting_proof.
3. عرض صفحة التعليمات:
   - رقم المحفظة / حساب إنستاباي / IBAN (من إعدادات payments.manual_accounts).
   - المبلغ المطلوب بالضبط + رقم الطلب كمرجع في التحويل.
4. شاشة رفع إثبات التحويل (صورة) ← تُحفظ عبر Media Library وتُربط بالطلب.
5. الحالة تتحول إلى proof_under_review + إشعار للأدمن/مدير الطلبات.
6. المراجعة (البند 6):
   6a. مقبول ← paid ← processing + إشعار «تم تأكيد دفعك».
   6b. مرفوض ← payment_rejected + سبب + رابط إعادة الرفع ← يعود awaiting_proof.
7. صلاحية إعادة المحاولة N مرات، ثم إلغاء تلقائي بعد مهلة (72 ساعة بلا إثبات ← cancelled).
```

---

### 5.5 تدفق الدفع عند الاستلام (COD)

```
1. العميل يختار COD.
2. إنشاء الطلب بحالة awaiting_confirmation (وليس paid).
   - سبب: تقليل خطر رفض الاستلام ← إلزام تأكيد واتساب قبل التجهيز.
3. تأكيد الطلب عبر واتساب/اتصال (يدوي أو زر «تأكيد الطلب عبر واتساب»).
4. بعد التأكيد ← processing ← shipped (مع تحصيل COD لدى شركة الشحن) ← delivered.
5. رفض الاستلام ← cancelled + تسجيل السبب لمتابعة التكرار/الاحتيال.
```

## 6. سير عمل مراجعة إثبات الدفع (Payment Proof Review)

### 6.1 من يملك الصلاحية

| الفعل | الصلاحية المطلوبة | الأدوار |
|---|---|---|
| عرض إثبات الدفع | `payment_proof.view` | super_admin, admin, orders_manager |
| قبول/رفض الإثبات | `payment_proof.review` | super_admin, admin, orders_manager (+ support إن مُنحت صراحة، مقيّدة بمنتجاته) |
| تعديل حسابات التحويل | `payments.manual_accounts.manage` | super_admin, admin |

### 6.2 خطوات المراجعة

```
1. طابور المراجعة في Filament: كل الطلبات بحالة proof_under_review (بادج بالعدد + تنبيه).
2. المراجع يفتح الطلب ويرى: صورة الإثبات، المبلغ المتوقع، طريقة الدفع، بيانات العميل، رقم المرجع.
3. التحقق اليدوي: تطابق المبلغ + المُرسَل إليه + التاريخ + عدم تكرار نفس الإثبات على طلب آخر.
4. القرار:
   ✅ قبول ← الحالة paid ← إشعار العميل ← دخول التجهيز.
   ❌ رفض ← اختيار سبب (مبلغ ناقص / صورة غير واضحة / تحويل غير موجود / إثبات مكرر)
            ← الحالة payment_rejected ← إشعار العميل مع السبب ورابط إعادة الرفع.
5. تسجيل التدقيق (Audit): من راجع، متى، القرار، السبب — على كل طلب.
```

### 6.3 حواجز الأمان

- كل صور الإثبات **خاصة** (غير عامة) وتُقدَّم عبر روابط موقّعة مؤقتة (signed URLs) فقط.
- فصل الأدوار: المراجع لا يقبل طلبًا رفعه/راجعه حساب مشبوه نفسه.
- `support` (إن مُنح review) لا يرى إلا طلبات منتجاته المسموح بها.
- تفعيل `spatie/laravel-activitylog` على الطلبات والمراجعات والمستخدمين والإعدادات الحساسة.

### 6.4 الإشعارات المرتبطة

| الحدث | القناة | المستقبِل |
|---|---|---|
| رفع إثبات جديد | إشعار لوحة + بريد | admin, orders_manager |
| قبول الإثبات | واتساب/بريد | العميل |
| رفض الإثبات | واتساب/بريد + السبب | العميل |
| مرور 72 ساعة بلا إثبات | تذكير ثم إلغاء تلقائي (Cron `schedule:run`) | العميل + orders_manager |

## 7. ملاحظات تنفيذية ختامية (جاهزة للتطبيق)

1. **التوليد:** بعد بناء الـ Resources، وَلّد الصلاحيات ثم عدّل الإسناد في Seeder للأدوار السبعة أعلاه.
2. **السوبر أدمن:** `Gate::before` في `AppServiceProvider` — يدعم أكثر من حساب.
3. **نطاق الدعم:** عمود `allowed_product_ids` (JSON) على `users` + Policies (`OrderPolicy`, `ReviewPolicy`, `InquiryPolicy`) تجمع `hasPermissionTo` + شرط المنتج.
4. **إخفاء المالي عن IT:** عبر Policy على حقل `orders.view_financials` وإخفاء الأعمدة في جدول Filament.
5. **مفاتيح الدفع:** في `.env` أو جدول إعدادات مشفّر — لا في الكود؛ الوصول عبر `payments.settings` (super_admin + it فقط).
6. **الدفع اليدوي + COD هو المسار الافتراضي الذي يعمل دائمًا**؛ البوابة الأونلاين تُفعّل وتُخفى تلقائيًا حسب وجود المفتاح.
7. **التدقيق:** فعّل `spatie/laravel-activitylog` على الكيانات الحساسة.
8. **عزل اللوحة:** مسار `/admin` معزول عن واجهة الزائر، وكل دور يرى فقط الموارد المصرّح بها (يُخفى غير المصرّح تلقائيًا من التنقّل).
